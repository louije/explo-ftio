Point d'entrée obligatoire — Cette API est le point d'entrée obligatoire pour accéder aux données usager. Le jeton produit est volatile (durée de vie ~1 journée) et doit être transmis en header ft-jeton-usager aux autres APIs (diagnostic, statut, informations administratives, entretiens, rendez-vous, métiers recherchés).

Code retour8 valeurs — 3 succès · 5 erreurs

Le code retour détermine comment interpréter la réponse. Le préfixe est signifiant : S = succès métier (HTTP 200), R = erreur (HTTP 4xx/5xx). Seul S001 produit un jetonUsager exploitable.

Succès métier HTTP 200
  • S001 Usager trouvé — un jetonUsager est produit (topIdentiteCertifiee = O ou N)
  • S002 Aucun usager trouvé — pas de jeton (topIdentiteCertifiee = null)
  • S003 Plusieurs usagers trouvés — pas de jeton, résultat ambigu à lever en amont
Erreurs HTTP 4xx / 5xx
  • R997 Erreur de validation des paramètres (HTTP 400)
  • R001 Accès non autorisé (HTTP 403)
  • R002 L'usager n'est pas rattaché à la structure de l'accédant (HTTP 403)
  • R998 Un service a répondu en erreur (HTTP 500)
  • R999 Service indisponible, à réessayer ultérieurement (HTTP 503)
ResultatDTO.codeRetour — également présent dans les 4 DTO d'erreur. Le champ message porte la version humaine du code. Non déclaré en enum : la liste est reconstituée uniquement à partir des exemples de réponses de l'OpenAPI.

Même usager, même jeton

S001 — Usager trouvé

Les deux méthodes produisent le même jeton pour le même usager. Le jeton peut ensuite être transmis en header ft-jeton-usager à toute API du réseau.

Cycle de vie du jeton : le jeton est volatile, valide environ une journée. Au-delà, il faut relancer une recherche pour en obtenir un nouveau. Le jeton ne contient aucune donnée personnelle en clair — c'est un identifiant crypté opaque.
