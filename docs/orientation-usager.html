Prérequis — Cette API est dans le royaume /agent (client credentials). Chaque appel nécessite le header ft-jeton-usager obtenu via l'API Rechercher usager, ainsi que les trois headers d'identification agent : pa-nom-agent, pa-prenom-agent, pa-identifiant-agent.
Flux d'orientation (deux acteurs) — 1. L'agent d'origine (ex. CAF, CD) appelle calculOrientation — attention, cet appel a des effets de bord : il calcule le parcours et enregistre immédiatement l'orientation dans le système (statut PRECONISE, état EN_ATTENTE). 2. L'organisme d'accueil (ex. Mission locale) découvre l'orientation via lectureOrientation filtrée sur etatOrientation=EN_ATTENTE. 3. Il accepte ou refuse via decisionOrientation → l'orientation passe à DECIDE/OUVERT. Il n'y a pas d'endpoint de notification : c'est l'enregistrement par calculOrientation qui rend l'orientation visible à l'organisme destinataire. L'endpoint POST /orientation permet d'enregistrer une orientation directement (sans calcul de critères).

Deux acteurs, un parcours — Un allocataire RSA s'inscrit via la CAF. L'agent CAF saisit les critères et lance le calcul, qui enregistre immédiatement une préconisation. Le lendemain, la Mission locale consulte ses orientations en attente, découvre la préconisation, et l'accepte. En parallèle, un conseiller départemental oriente directement un autre usager sans passer par le calcul.
